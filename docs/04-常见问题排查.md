## 问题1 — HDFS安全模式导致上传阻塞

### 现象

```bash
$ hdfs dfs -mkdir /car_complaint/input
# 命令执行了但目录没创建

$ hdfs dfs -put file.csv /car_complaint/input/
# 卡顿或失败
```

### 原因

NameNode 启动后进入 **Safe Mode（安全模式）**，只允许读操作，禁止写操作。这是 HDFS 的保护机制，确保 DataNode 上报足够的块报告后才能开始服务。

### 排查

```bash
$ hdfs dfsadmin -safemode get
Safe mode is ON in node1/192.168.88.101:9000
```

### 解决

```bash
# 方案A：等待30秒自动退出
sleep 30

# 方案B：手动退出
hdfs dfsadmin -safemode leave

# 验证
hdfs dfsadmin -safemode get
# Safe mode is OFF
```

## 问题2 — YARN cluster模式任务不断ACCEPTED

### 现象

```bash
$ spark-submit --master yarn --deploy-mode cluster ...
26/06/22 10:40:40 INFO Client: state: ACCEPTED
26/06/22 10:40:41 INFO Client: state: ACCEPTED
# ... 持续ACCEPTED，永远不RUNNING
```

### 排查

```bash
$ yarn application -list
# application_xxx_0001: state=ACCEPTED

$ yarn node -list -all
# 3个节点 RUNNING，每节点8GB/8核，Container=0
# 资源充足但调度器不分配
```

### 解决

对于本项目的 **小数据量场景（4,700条）**，改用 local 模式：

```bash
spark-submit \
  --class car.complaint.analysis.ComplaintDataClean \
  --master local[2] \     # 本地2线程，绕过YARN
  car_complaint_analysis-1.0-SNAPSHOT.jar
```

> `local[2]` 使用本机2个线程并行计算，任务在10秒内完成。对于GB级大数据才需要YARN。

## 问题3 — JDK 21 编译 Scala 2.12 报错

### 现象

```bash
$ mvn clean package

error: scala.reflect.internal.FatalError:
  bad constant pool index: 0 at pos: 48462
     while compiling: <no file>
```

### 原因

JDK 21 生成的 `.class` 文件常量池格式与 Scala 2.12 编译器**不兼容**。Scala 2.12 官方支持的 JDK 版本范围是 **8 ~ 11**。

### 解决

在 IntelliJ IDEA 中指定 JDK 8 作为编译 SDK：

1. `文件 → 项目结构 → SDK` → 添加 JDK 8 路径
2. `文件 → 设置 → Maven → 导入程序的JDK` → 选 JDK 8
3. 重新 `mvn clean package`

> **检查本机已装 JDK 1.8**：`java -version` 或 `where java`

## 问题4 — MariaDB 连接超时

### 现象

Spring Boot 启动后报错：

```
java.sql.SQLNonTransientConnectionException:
Socket fail to connect to host:address=(host=192.168.88.101)(port=3306)(type=primary).
Connection timed out: getsockopt
```

### 排查 — 逐层定位

```bash
# 第1层：IP 通不通？
C:\> ping 192.168.88.101
# 通：<1ms → 网络层正常

# 第2层：端口通不通？
C:\> telnet 192.168.88.101 3306
# 不通：连接失败 → 传输层被防火墙阻挡
```

| 层 | 测试工具 | 本法项目 |
|----|----------|----------|
| IP层 | ping | ✔ 通 |
| TCP层 | telnet | ❌ 不通 → 防火墙 |
| 应用层 | mysql | - |

### 解决

在虚拟机 node1 上开放 3306 端口：

```bash
# 方案A：添加防火墙规则
sudo firewall-cmd --add-port=3306/tcp --permanent
sudo firewall-cmd --reload

# 方案B：关防火墙（仅实训环境）
sudo systemctl stop firewalld
```

验证：

```bash
C:\> telnet 192.168.88.101 3306
# 出现乱码字符（MariaDB协议握手包）→ 连通！
```

## 问题5 — ECharts 热力图数据格式转换

### 现象

品牌×问题交叉API返回：

```json
{
  "brand": ["比亚迪","大众",...],
  "category": ["车身","电气设备",...],
  "data": [[15,3,0,...], [8,12,5,...], ...]
}
```

但 ECharts heatmap 需要：

```json
// [colIndex, rowIndex, value] 三元组
[[0,0,15], [1,0,3], [0,1,8], ...]
```

### 解决 — 前端转换代码

```javascript
const topBrands = data.brand.slice(0, 15);  // 取TOP15品牌
const heatData = [];

topBrands.forEach((brand, bi) => {
  data.category.forEach((cat, ci) => {
    const val = data.data[bi]?.[ci] || 0;  // 可选链安全取值
    if (val > 0) heatData.push([ci, bi, val]);  // 只保留非零值
  });
});

// 然后 heatData 就是 ECharts 需要的格式
```

## 问题6 — Spark CSV header 被 MariaDB 误读为数据行

### 现象

Spark 输出 CSV 时带了 header：

```csv
name,cnt,problem_category_count,problem_type_count
比亚迪,613,17,63
```

MariaDB LOAD DATA 导入后：
```
| name                    | cnt     |
|-------------------------|---------|
| name                    | 0       |  ← header 被当成数据了！
| 比亚迪                  | 613     |
```

### 解决

Spark 代码中去掉 `.option("header", "true")`：

```scala
// ❌ 错误
result.write.mode(SaveMode.Overwrite)
  .option("header", "true")   // 删除这行
  .csv("...")

// ✔ 正确
result.write.mode(SaveMode.Overwrite)
  .csv("hdfs://node1:9000/car_complaint/output/BrandAnalysis")
```

对于有自增主键的表（brand_problem_cross），导入时显式指定列：

```sql
LOAD DATA LOCAL INFILE '/tmp/cross.csv'
INTO TABLE brand_problem_cross
FIELDS TERMINATED BY ',' LINES TERMINATED BY '\n'
(brand, problem_category, cnt);  -- 跳过 id 列
```
